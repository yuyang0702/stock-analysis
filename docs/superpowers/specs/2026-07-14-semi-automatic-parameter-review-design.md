# 半自动参数复核与版本化发布设计

> 主文档：`docs/project_roadmap.md`。本文从属于主文档，并细化 `docs/superpowers/plans/2026-07-13-layered-exit-risk-management.md` 的 Batch G。状态冲突时以主文档为准。

## 1. 状态与目标

本文记录的是 `planned` 设计，不代表代码已实现、服务器已部署、模拟盘已观察或策略已验证。

当前可复用基础：

- `ml_dataset.py` 已采集信号、订单状态和部分收益标签。
- `strategy_compare_report.py` 已输出原策略与规则型影子评分对照。
- `backtest_engine.py` 已支持对既有信号样本做第一版信号级回测。
- 独立逐日历史回测框架已在本地 `implemented`，但尚无真实 strict 数据运行和验证证据。
- SQLite 策略运行记录已有 `parameter_version` 字段。

当前尚不存在：

- 可审计的参数候选登记表。
- 基线与候选的统一离线评价器和准入门。
- 人工批准、拒绝、激活和回滚状态机。
- 自动分析定时任务和参数复核报告。
- 任何会自动改变模拟盘或实盘参数的学习系统。

目标是在数据充足后自动生成候选和证据，由用户明确批准后才允许把某个不可变参数版本同步到 JoinQuant 模拟盘；系统不得自行批准、发布、部署、重启或接触真实资金。

## 2. 决策

采用“自动分析，人工批准，显式发布，可一键回滚”的半自动方案：

```text
冻结基线参数
→ 收集真实模拟盘证据
→ 自动生成少量候选
→ 时间切分回测和分层评价
→ 输出准入结论与风险说明
→ 人工按候选ID和内容哈希批准或拒绝
→ 单独授权后发布到模拟盘
→ 观察、比较、必要时回滚
```

人工批准不是形式确认。批准记录必须绑定候选 ID、父版本、规范化参数内容哈希、评价窗口和评价结果；候选内容变化后原批准立即失效。

不采用以下方案：

- 只生成一份无版本、无门槛的调参建议：无法审计，也容易根据单日盈亏追参数。
- 自动通过即自动发布：越过用户权限边界，且执行故障、数据泄漏和过拟合会直接进入交易链路。
- 强化学习直接控制买卖：当前样本、回测和执行账本均不满足前置条件。

### 2.1 与机器学习路线的关系

本设计治理的是确定性策略参数，不把“搜索候选参数”包装成已经训练机器学习模型。首版候选生成采用受白名单限制的确定性搜索，不引入训练框架或模型依赖。

未来训练模型属于主路线图 ML-7：模型必须拥有独立的模型版本、训练代码与特征版本、时间切分和验证/保留集证据，先以影子 `ml_score` 运行。模型可以提出排序建议或参数研究候选，但不能写入 `approved / promoted / rolled_back` 等治理状态，也不能直接修改活动参数。

信号审计必须同时区分代码版本、模型版本和参数版本。没有训练模型时，模型版本明确记录为 `none` 或规则型影子评分版本，不得把 `shadow_score.py` 的规则增强分描述为已训练模型。

## 3. 参数权限分层

### 3.1 永久硬约束

以下能力不属于参数学习器的可调范围，候选生成器不得提出关闭、放宽或绕过它们的方案：

- `KILL_SWITCH`、是否允许卖出和健康门的卖出安全降级。
- T+1、停牌、涨跌停、100 股单位、可卖数量和未完成委托约束。
- 单票、行业、题材、总仓位和组合开放风险的绝对安全上限。
- 硬止损必须存在、初始止损不得下移、卖出优先级高于买入。
- 陈旧行情、异常价格、重复委托、幂等和账本一致性保护。
- 认证、服务器地址、令牌、通知、部署和运维配置。

硬约束只能通过独立设计、代码评审和用户明确授权修改，不能由参数复核流程更改。

### 3.2 可复核参数族

首版只允许在预先定义的安全区间内提出候选，每个候选最多改变一个参数族：

- 初始止损距离中的板块/ATR系数，但不能取消硬止损或突破最大损失上限。
- 首段止盈 R 倍数、剩余仓位比例和移动止盈 ATR 系数。
- 短线/中线时间止损天数与最低进展阈值。
- `NORMAL / CAUTION` 下的风险预算，但不能超过硬编码绝对上限，`RISK_OFF` 仍禁止新买。
- 买入分数门槛、追价容忍和退出后冷却期等非安全底线参数。

候选范围、步长、默认值和绝对上下界必须进入代码中的版本化白名单，并由测试锁定。环境变量只能选择已批准版本或实施紧急回滚，不得成为绕开登记表的隐式调参通道。

## 4. 数据就绪与候选生成

### 4.1 报告就绪门

未同时满足以下条件时，只输出“数据不足”报告，不生成可批准候选：

- 至少 20 个有效交易日。
- 至少 30 个已闭合持仓周期，且订单、成交、费用和退出原因可以关联。
- 拟调整参数族至少有 30 个相关触发样本；任何单独分层不足 10 个样本时不得对该分层下结论。
- 无未关闭 P0/P1 数据一致性、重复委托、成交关联或数据库完整性问题。
- 基线参数版本在评价窗口内冻结；窗口中途变更时从新版本重新计数。

20 个有效交易日只是开始复核的最低门槛，不等于候选可以发布。

### 4.2 发布就绪门

候选进入人工批准列表前，还必须具备以下任一证据组合：

1. 真实 strict 数据通过完整历史回测质量门，候选通过至少 3 个不重叠的 walk-forward 验证窗口，并有至少 20 个有效模拟盘交易日作为现实校准；或
2. 缺少可用的完整历史回测证据时，累计至少 60 个有效模拟盘交易日，并使用前段生成候选、中段验证、末段保留集确认，保留集在候选确定前不可查看。

如果两类证据均不满足，系统可以生成研究候选，但状态只能是 `research_only`，不得进入 `approvable`。

### 4.3 候选搜索约束

- 每次复核最多输出 5 个候选，每个参数族最多 2 个。
- 每个候选只改变一个参数族，避免无法归因的组合搜索。
- 搜索空间由白名单和有限步长构成，不使用无界网格、随机搜索或反复试探保留集。
- 同一数据窗口和同一代码版本下重复运行必须得到相同候选 ID、内容哈希和结论。
- 候选按净收益、回撤、尾部损失、Profit Factor、成交成本和稳定性综合排序，不以单一胜率或总收益决定。

## 5. 评价与准入门

所有收益指标均按实际可成交数量、手续费和滑点计算。至少比较：

- 净收益、年化口径仅作为展示，不夸大短样本。
- 最大回撤、单笔最大亏损和尾部损失。
- Profit Factor、盈亏比、胜率和闭合周期数。
- 换手、滑点、拒单、部分成交和退出延迟。
- 按策略模式、市场状态、行业/题材、分数区间和持仓天数分层的结果。

候选必须同时通过：

1. 至少 2/3 walk-forward 窗口优于基线，保留集不劣于基线。
2. 扣除费用后的净收益不低于基线，Profit Factor 至少相对提升 5%。
3. 最大回撤相对恶化不超过 10%，且绝对恶化不超过 1 个百分点；任一硬风险指标越界直接拒绝。
4. 去掉收益最高的 3 笔交易后，候选相对基线的收益差仍不为负，避免由少数离群交易驱动。
5. 任何样本充足的关键分层都不得出现灾难性退化；定义为净收益转负且最大回撤恶化超过 1 个百分点，或执行失败率翻倍。
6. 不存在未来数据、幸存者偏差、重复样本、参数版本混用或无法解释的数据缺口。

门槛数值是首版冻结基线。修改门槛本身属于策略治理变更，也需要版本、测试和人工批准，不能因为候选未通过而临时放宽。

## 6. 版本与状态机

参数版本采用不可变标识，例如 `risk-2026-07-14.1`，并保存父版本和规范化 JSON 的 SHA-256。

候选状态：

```text
research_only
→ approvable
→ approved | rejected | expired
→ promoted
→ observing
→ validated | rolled_back | retired
```

状态规则：

- 自动任务最多把候选推进到 `approvable`。
- `approved / rejected` 必须由用户显式操作，记录操作时间、候选哈希和理由。
- 批准只代表允许发布，不代表已经 `deployed`。
- 发布必须是独立的、可审计的实施动作；完成服务器和 JoinQuant 同步后才是 `deployed`。
- 至少产生一个有效模拟盘交易日证据后才是 `observed`。
- 达到专项观察门槛并通过人工验收后才是 `validated`。
- 父版本、代码版本、数据窗口或候选内容变化时，批准失效并回到复核流程。

任一时间只允许一个活动参数版本。激活和回滚都使用比较并交换：命令必须提供预期当前版本和目标内容哈希，防止并发或误操作覆盖新版本。

## 7. 发布、观察与回滚

首版只面向 JoinQuant 模拟盘。批准后仍需要用户明确授权执行代码提交、推送、服务器拉取、参数激活、网站模板同步或服务重启；历史授权不得复用。

发布前：

- 备份 SQLite，并验证恢复副本、schema 和完整性。
- 记录服务器 Git SHA、JoinQuant 模板版本、活动参数版本和目标哈希。
- 执行完整测试、Linux 专项测试和 dry-run 参数解析。
- 确认回滚版本仍可读取且没有被环境变量覆盖。

发布后：

- 首个有效交易日逐笔核对参数版本、信号、目标数量、委托、成交和退出原因。
- 连续 3 个有效交易日验证执行安全；执行错误立即回滚，不等待收益样本。
- 至少 20 个有效交易日评价策略表现，不因单日或少数交易亏损回滚或再调参。
- 每 20 个有效交易日最多批准并发布一个参数版本，避免连续试错污染证据。

立即回滚条件：参数版本不一致、硬约束被绕过、错误买卖、重复有效委托、数据泄漏、账本损坏或结果无法复算。表现回滚必须按冻结门槛和完整窗口判断，不能凭单日盈亏决定。

## 8. SQLite 与文件设计

以下为 `planned` 表，不属于当前 schema version 6 的实际范围：

- `parameter_sets`：不可变参数内容、版本、父版本、代码版本、哈希和状态。
- `parameter_evaluations`：数据窗口、样本计数、分层指标、门槛结果和评价器版本。
- `parameter_decisions`：人工批准/拒绝、候选哈希、理由和时间。
- `parameter_activations`：激活、观察、验证、回滚和退役事件。

要求：

- 显式 schema migration、事务、外键、唯一约束和常用查询索引。
- 每次复核最多 5 个候选；评价明细保存聚合指标和必要引用，不复制行情、订单或全部交易明细。
- SQLite 是长期审计事实源；报告不是审批事实源。
- `output/parameter_review_latest.md` 使用原子覆盖，只表示最近一次结果。
- 只有候选状态变化或阶段验收时生成低频归档报告；不新增逐轮 JSONL 或每次扫描文件。
- 以每周一次复核、每次 5 个候选和每候选不超过 100 KB 聚合结果估算，新增 SQLite 数据低于约 30 MB/年；超过 50 MB/年或单次评价超过 5 MB 时必须告警并重新评审设计。
- 沿用交易账本在线备份、完整性检查和恢复演练；这些能力虽已本地实现，但在服务器部署、观察和恢复验收前，不得把本能力标为可发布。

## 9. Codex 与自动化权限边界

Codex 定时审核员可以只读：

- 参数复核最新报告和候选聚合指标。
- 候选、评价、决策和激活记录的只读查询结果。
- 活动版本、服务器 Git SHA、JoinQuant 模板版本和一致性证据。

Codex 可以说明某候选为什么通过或失败、指出数据不足和提出独立设计建议，但不可以：

- 创建、修改或删除候选和评价记录。
- 写入批准、拒绝、激活、回滚或验证状态。
- 修改参数文件、环境变量、SQLite、代码或文档。
- 提交、推送、部署、重启服务或操作订单。

本地自动任务依赖电脑开机、联网、Codex 可运行和服务器只读 SSH 可用。即使任务拥有服务器只读权限，也不得获得参数发布或运维写权限。

## 10. 验收标准

### implemented

- 全部新表、评价器、报告、批准/发布命令和保护测试完成。
- 同一输入可复现相同候选与哈希；未来数据和保留集泄漏测试失败时硬拒绝。
- 未批准、哈希变化、活动版本不匹配和硬约束越界均无法激活。
- 不生成无界文件，增长、备份和恢复测试通过。

### deployed

- 服务器运行包含本能力的 Git 版本和目标 SQLite schema。
- 自动分析 timer 只生成报告，不改变活动参数。
- JoinQuant 和服务器显示相同活动参数版本；发布权限与只读审核权限隔离。

### observed

- 至少一个有效交易日产生可复算的候选或活动版本证据。
- 报告、SQLite、信号、订单和快照中的参数版本一致。

### validated

- 自动分析连续稳定运行至少 20 个有效交易日且无越权写入。
- 至少完成一次“生成候选—人工决定—模拟盘发布或拒绝—观察/回滚”的审计演练。
- 候选评价可由独立命令复算，备份可恢复，权限边界经人工确认。

策略收益通过门槛与系统流程通过门槛必须分别报告；流程 `validated` 不代表某个候选一定优于市场或未来仍有效。
