## 1. 目标与边界

目标是回答：“过去6个月或1年，如果每天只使用当时已经公开且可获得的数据运行当前策略，扣除费用、滑点和A股撮合限制后，收益、回撤与稳定性如何？”

本阶段实现可重复、可审计的逐日事件回测框架、数据导入校验、严格/代理双轨运行、walk-forward评价和参数对照。它不自动批准参数，不改活动策略，不写运行交易账本，不连接真实资金，也不把代理回测冒充完整策略验证。

现有 `backtest_engine.py` 保留为“既有信号级回测”，兼容原CLI与输出。逐日历史重跑使用独立模块和独立入口，避免把两种证据混在同一报告中。

## 2. 双轨定义

### 2.1 `strict` 严格时点模式

每个交易日的选股、评分、市场环境、题材、行业、公告/新闻和交易约束只能读取 `available_at <= decision_at` 的记录。所有启用的策略因子必须在数据清单中声明来源、字段、发布时间口径、复权口径和覆盖率。

下列任一情况硬拒绝运行或拒绝对应窗口，不允许静默填零：

- 股票池没有按日期保存，导致退市股或后来上市股票引入生存者偏差；
- 非价格因子缺少发布时间或只能获得当前值；
- 复权方式、前收盘、涨跌停、停牌或ST状态不明确；
- 同一证券同一交易日出现冲突行情；
- 交易日、参数版本、策略版本或数据版本混用；
- 窗口覆盖率低于配置门槛；默认行情与交易状态必须100%，启用因子必须100%。

只有 `strict` 模式完整通过数据质量门，才能标为“完整历史回测证据”。代码实现仍只表示 `implemented`；实际导入6个月/1年数据并成功运行是 `observed`，通过人工复核和重复运行一致性后才是 `validated`。

### 2.2 `price_core` 价格核心代理模式

只使用历史日线、当日股票状态和可从当时及以前行情确定性计算的技术/风险字段。新闻、龙虎榜、题材热度、当前行业归属等缺失因子不猜测、不用未来值替代，并在清单与报告中列为 `excluded_features`。

该模式用于验证价格技术、仓位、分层退出、费用和撮合敏感性。无论测试结果多好，报告固定标识 `proxy_only=true`，结论不得升级为完整策略通过，不得满足Batch G“完整历史回测”准入门槛。

## 3. 最小架构

采用现有Python、pandas和SQLite，不新增依赖。

### 3.1 `historical_data.py`

负责独立历史库 `cache/backtest/history.db` 的建库、CSV导入、数据哈希、覆盖率和时点检查。它不联网下载；数据来源由适配器或人工导出提供，首版支持JoinQuant/AkShare标准CSV映射。

核心表：

- `dataset_manifests`：数据集ID、来源、版本、复权口径、创建时间、文件SHA-256、起止日期、覆盖率和声明字段；
- `daily_bars`：交易日、证券、开高低收、前收、成交量、成交额、复权因子；
- `daily_status`：上市、ST、停牌、涨停价、跌停价和当日是否可交易；
- `daily_universe`：每个交易日实际可见股票池；
- `point_in_time_features`：严格模式额外因子，包含特征名、值、事件时间和 `available_at`；
- `backtest_runs`：运行配置、代码/策略/参数/数据版本、结果摘要和输入哈希；
- `backtest_equity`、`backtest_trades`：独立保存逐日权益和逐笔模拟成交。

主键和唯一约束拒绝重复冲突数据；相同输入文件重放幂等。历史库与正式 `trading.db` 完全分离。

### 3.2 `historical_strategy.py`

提供纯函数接口：

```python
generate_daily_candidates(
    trade_date: str,
    universe: list[dict],
    history: dict[str, list[dict]],
    features: dict[str, dict],
    *,
    mode: str,
    parameter_version: str,
) -> list[dict]
```

首版复用已有可独立调用的评分、初始止损、市场状态、风险定仓和退出函数；不能证明时点安全的实时抓取、新闻聚合或当前快照函数不得从回测路径调用。`strict` 缺字段即返回稳定错误码，`price_core`只运行已声明的价格核心因子。

### 3.3 `historical_backtest.py`

事件循环按交易日执行：

```text
T日开盘：处理上一交易日产生且仍有效的订单
T日日内：用OHLC确定止损/止盈是否触发
T日收盘：按T日及以前数据生成下一交易日订单
T日收盘后：盯市、记录权益和质量统计
```

默认使用“T日收盘决策、T+1开盘成交”，避免用当日收盘价同时生成并成交信号。可选 `same_close` 仅用于敏感性对照，报告必须单独标识，不作为默认结论。

## 4. 撮合与账户规则

- 买卖数量按100股整数单位；
- 买入佣金默认万三且最低5元；卖出另收千一印花税；
- 滑点按成交价基点配置，默认买卖各10bp；
- A股买入当日不可卖，卖出仅使用可卖数量；
- 停牌不成交；开盘价达到或越过涨停时买单不成交，达到或越过跌停时卖单不成交；
- 订单未成交不会虚构成交，按配置决定当日失效或下一日重试，默认当日失效；
- 单票、总仓位、开放风险、行业/题材和现金约束使用运行时参数版本；
- 先处理卖出再处理买入；同日多个买入按确定性分数、代码排序分配现金；
- 止损缺口按可成交的下一价格处理，不假设能以止损价成交；
- 同一根日线同时触发止损与止盈时采用保守顺序：先止损；报告统计这一保守假设次数；
- 分红送转首版只通过未复权价格加复权因子保持数量和成本连续，不单独模拟现金红利；缺复权因子时严格模式拒绝跨除权窗口。

## 5. 防未来函数与数据质量

所有查询必须显式传入 `as_of_date` 或 `decision_at`。回测代码禁止读取“最新文件”、系统当前日期、正式运行SQLite、当前持仓、当前行业缓存和实时网络接口。

每次运行先生成质量报告并计算：

- 股票池、日线、交易状态和启用因子覆盖率；
- 重复、冲突、日期倒序、负价格、OHLC不合法和缺失前收数量；
- `available_at > decision_at` 的未来数据数量；
- 上市前、退市后或停牌日错误信号数量；
- 数据集、代码、参数、策略和配置的联合SHA-256。

只要严格质量门失败，就不生成收益结论，只输出 `rejected` 和稳定原因码。

## 6. Walk-forward与参数对照

6个月或1年区间按时间顺序切分，至少产生3个互不重叠的验证窗口。训练/候选生成区只允许使用验证窗口之前的数据；最后保留集在候选固定前不可参与排序。

基线和候选必须共享同一数据集、窗口、撮合、费用、滑点和初始资金。每个候选只改变一个参数族；本阶段只提供评价结果，不写 `approved`、`active` 或部署状态。

每个窗口和总体输出：

- 扣费净收益、年化收益、最大回撤、波动率和Calmar；
- 胜率、平均盈亏、Profit Factor、尾部5%交易损失；
- 交易数、换手率、平均持有期、未成交/拒单原因；
- 按策略模式、市场环境、分数区间和可用题材/行业分组；
- 去掉最高3笔盈利后的净收益与Profit Factor；
- 费用、滑点、涨跌停和同柱冲突的敏感性结果。

## 7. CLI与输出

新增独立入口，不改变现有 `backtest`：

```bash
python historical_backtest.py import --source joinquant --input data.csv --db cache/backtest/history.db
python historical_backtest.py validate --mode strict --start YYYY-MM-DD --end YYYY-MM-DD
python historical_backtest.py run --mode strict --start YYYY-MM-DD --end YYYY-MM-DD
python historical_backtest.py run --mode price_core --start YYYY-MM-DD --end YYYY-MM-DD
python historical_backtest.py compare --baseline baseline.json --candidate candidate.json
```

`run_ubuntu.sh` 增加 `historical-backtest` 和 `historical-backtest-validate`，不挂定时器。

固定输出采用原子覆盖：

- `output/historical_backtest_latest.md`；
- `output/historical_backtest_quality.json`；
- `output/historical_backtest_equity.csv`；
- `output/historical_backtest_trades.csv`；
- `output/historical_backtest_compare.json`。

历史运行事实以独立历史库为准，报告不是审批事实源。

## 8. 存储增长与保留

一年约250个交易日、5000只证券，日线和状态约250万行。独立历史库年度目标低于2GB；导入前后记录文件大小和行数。单次导入增长超过3GB、重复率超过0.1%、质量报告超过20MB或单日异常增长超过100MB时拒绝继续导入。

原始导入文件由用户决定是否保留；数据库不自动删除。运行明细保留最近20次完整结果和所有被人工标记的基线/候选证据；首版不自动清理，由后续独立设计处理。历史库不进入正式交易账本每日备份，需单独备份或可从带哈希的源数据重建。

## 9. 失败处理

- 导入在单事务中完成，失败不留下半个数据集；
- 运行写入临时run状态，成功后标记 `complete`，异常标记 `failed` 并保留有界错误摘要；
- 输入哈希相同的完整运行可复用结果，配置或代码版本变化必须产生新run ID；
- 数据缺口、未来数据、版本混用和冲突行属于硬错误；
- 网络下载、自动补数和自动修复不属于首版，避免不可重复和静默漂移。

## 10. 测试与验收

自动化测试必须覆盖：

- CSV导入幂等、冲突回滚、数据哈希与覆盖率；
- 严格模式缺字段/未来字段拒绝，代理模式明确列出排除因子；
- 无生存者偏差的按日股票池；
- T+1、停牌、涨停买拒绝、跌停卖拒绝、100股单位、费用和滑点；
- 次日开盘成交、缺口止损、同柱止损优先、卖出优先和现金确定性分配；
- 分层退出、仓位和开放风险；
- 复权因子缺失拒绝；
- 3个不重叠walk-forward窗口与保留集隔离；
- 基线/候选同口径、输入乱序结果一致；
- Profit Factor、回撤、尾部损失和去除最高3笔收益可人工复算；
- 运行重放幂等、原子输出和正式交易账本零写入；
- 原信号级回测全部无回归。

验收只允许以下状态：

| 状态 | 条件 |
|---|---|
| `planned` | 只有本设计和实施计划。 |
| `implemented` | 代码与平台无关测试通过，尚无真实完整数据运行。 |
| `deployed` | 指定环境具备历史库、CLI和版本一致代码；不代表回测有效。 |
| `observed` | 至少一个6个月或1年数据集通过质量门并成功重复运行。 |
| `validated` | 严格模式、至少3个walk-forward窗口、人工复算和重复性检查全部通过。 |

`price_core` 永远只能标记为代理研究证据，不能单独达到“完整历史回测 validated”。
