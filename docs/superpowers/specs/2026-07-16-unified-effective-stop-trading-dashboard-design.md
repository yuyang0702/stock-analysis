# 统一有效止损与交易运行面板设计

日期：2026-07-16  
状态：基础能力 implemented / deployed / not observed / not validated；可观测性增强 implemented / not deployed / not observed / not validated
从属关系：从属于 `docs/project_roadmap.md`；卖出优先级继续从属于分层退出设计，本文只统一止损事实源、成交后校验与网页展示/维护边界。

## 目标

解决网页按成本价自动生成 3.5% 止损、而执行引擎读取持仓周期冻结止损所造成的双重事实源。完成后，网页和策略必须调用同一个纯函数解析唯一卖出阈值，并把运行状态、异常、持仓风险和执行轨迹集中展示在手机端交易运行面板。

## 状态词

- `planned`：本文确认但尚未进入代码。
- `implemented`：代码、迁移和测试完成。
- `deployed`：服务器运行该提交且数据库迁移、服务重启和配置校验完成。
- `observed`：有效交易日产生真实快照、信号、委托或成交证据。
- `validated`：满足主文档要求的连续观察日和人工验收门槛。

部署成功不能自动提升为 observed 或 validated。

## 唯一止损模型

`position_cycles` 是活动持仓止损事实源：

- `initial_stop_price`：入场信号止损按真实成交均价重新校验后的冻结初始止损。多头取 `max(signal_stop, fill_guard)`，其中 `fill_guard = fill_price * (1 - board_max_loss)`；主板低波动 6%、主板活跃 7%、创业板/科创板 9%。结果不得高于成交价减 0.01，也不得在后续快照中下移。
- `manual_stop_price`：用户在网页明确设置的可选止损，默认 `NULL`。首版只允许新增、上调或清除；禁止下调，避免人工操作放宽既有风险边界。每次变化写入 `control_events`，包含操作者、旧值、新值、原因和时间。
- `trailing_stop_price`：派生值，不重复持久化。只在 `take_profit_stage >= 1` 且 ATR 有效时激活。短线为最高价减 2 ATR，中线为最高价减 3 ATR；RISK_OFF 各收紧 0.5 ATR，最低 1.5 ATR；不得低于初始止损。
- `effective_stop_price`：唯一卖出阈值，不持久化重复副本。取已生效的 `initial_stop_price`、`manual_stop_price` 和 `trailing_stop_price` 最大值。网页与卖出引擎必须调用同一解析器。

当现价小于等于有效止损时生成/续驱退出意图；跳空按真实可成交价成交。T+1、停牌、跌停或可卖数量不足只改变执行阶段，不得把退出意图视为完成。卖出不受停买状态影响。

## 旧持仓迁移

schema version 8 新增 `manual_stop_price`。迁移不伪造成交、不覆盖历史信号、不把旧网页自动 3.5% 值冒充用户手工指令。活动周期在下一次新鲜 JoinQuant 快照对账时，用真实持仓成本和对应板块最大亏损校验冻结初始止损：只允许上调，不允许下调；变化写入 `control_events`。缺少可靠成本或股票代码时保持 `legacy_fixed` 并在面板提示人工核验。

因此已知异常持仓 `301379` 不会把网页旧显示值 36.18 自动写成 manual stop；它会按真实成本应用创业板 9% 成交保护下限，并在下一交易轮按统一有效止损判断。只有用户随后明确设置手工止损，才产生 manual stop。

## 网页边界

网页改名为“交易运行面板”，移动端优先，包括：

1. 运行状态：交易阶段、最近扫描/快照、买卖许可、kill switch、最近对账、代码版本和数据陈旧状态。
2. 异常：活动 `execution_issue_state`、触发止损但未完成卖出、信号/订单/成交问题、市场阻塞和止损数据不完整。
3. 持仓风险：成本、现价、盈亏、数量/可卖、初始/手工/移动/有效止损、距离、退出阶段和更新时间。
4. 执行轨迹：该证券近期退出意图、订单事件、成交和对账问题。
5. 手工维护折叠区：只维护手工止损和备注，不提供直接买入、卖出或清仓按钮。

截图上传、OCR、识别确认、截图止盈止损默认值、上传文件访问路由和 OCR 专用依赖代码全部移除。历史上传文件保留，不由迁移自动删除。

## 网页安全

网页使用 `PORTFOLIO_WEB_TOKEN`；未单独配置时为兼容部署复用现有 `JOINQUANT_SYNC_TOKEN` 的值，但不修改该值。凭据只通过 POST 登录或 Authorization header 传输，不再放入 URL。认证成功后使用 HttpOnly、SameSite=Strict 的签名会话 cookie；所有状态修改使用 CSRF token。日志、页面、异常和审计不得输出认证 token。登录失败、CSRF 失败和手工止损变更均留下不含密钥的有界审计证据。

JoinQuant API 优先接受 `Authorization: Bearer`，在模板迁移期间保留 query token 兼容，但服务器事件日志不得保存 token 或完整查询串。

## 存储与增长

本设计只给现有 `position_cycles` 增加一个可空列，复用 `control_events` 和 `execution_issue_state`；不新增逐扫描 JSONL、截图或重复有效止损列。历史上传目录只读保留。账本、备份、366 天高频保留和 7/4/12 备份轮转规则不变。

2026-07-19 可观测性增强不新增表、列、数据库、JSONL、截图、逐刷新事件或永久报告，
不改变 SQLite schema、保留期或备份轮转。风险指标从现有活动周期和有限持仓快照派生；
数据不足时显示未知，不为展示目的迁移 schema。

## 验收

- 同一状态输入下网页与策略解析出的 effective stop 完全一致。

- OCR 路由和页面元素返回 404/不存在，历史文件不删除。
- 停买或 kill switch 状态下，合法卖出仍可生成和续驱。
- 部署前后环境文件哈希一致，SQLite 备份完整，schema 8 健康可写，三个服务 active。
- 各数据源分别显示绝对时间、距今年龄和陈旧状态。
- 待执行订单和退出意图显示状态、等待时间及已知阻塞原因。
- 单只持仓可追溯信号、订单、成交、退出意图和对账差异。
- 能力状态严格区分 planned / implemented / deployed / observed / validated。
- 可观测性增强不新增持久化、直接交易、自动解锁、参数修改或模型启用入口。
