# 企业微信执行回报幂等与全量信号复盘设计

更新日期：2026-07-14

> 文档层级：本文件是企业微信成交回报、信号复盘和统一时间展示的专项设计从文档。
>
> 主文档：`docs/project_roadmap.md`。如项目状态、阶段或权限口径冲突，以主文档为准。
>
> 当前状态：`implemented（已推送、未部署） / not observed / not validated`。

## 1. 背景与已确认问题

2026-07-14 服务器只读证据确认了两个独立问题：

1. JoinQuant 每分钟账户快照持续携带同一笔已成交订单。服务器使用包含快照时间和总资产的完整 Markdown 摘要作为去重键，导致同一 `order_id` 每分钟得到不同去重键并反复推送。
2. 当前“推送跟踪复盘”只复盘仍出现在当轮候选结果中的最近信号，并最多展示 6 只。2026-07-13 成功推送的 4 条买点提醒没有进入 2026-07-14 复盘；当天复盘的 2 只均为当天新推送信号。

当前复盘存在幸存者偏差：掉出候选池的弱信号更容易被遗漏，不能形成可信的模拟盘策略验证样本。

## 2. 目标

- 执行回报由 SQLite 首次入账的真实成交事件驱动，而不是由周期账户快照中的历史订单列表驱动。
- 同一成交在重复快照、重复回调和服务重启后只产生一次逻辑通知。
- 部分成交新增时可以产生新的增量回报，不把合法增量误判为重复。
- D+1 完整复盘上一交易日所有成功推送的买点，不依赖股票是否仍在当前候选池。
- D+3、D+5、D+10 按交易日继续形成阶段复盘证据。
- 所有企业微信消息统一显示服务器发送时间。
- 复用现有 SQLite、`signal_watchlist.json` 和通知失败重试队列，不新增第三方依赖或第二套通知账本。

## 3. 非目标

- 不修改选股、评分、买点、卖点、仓位、止盈止损或 JoinQuant 撮合规则。
- 不改变订单、成交、账户快照和对账的正式账本语义。
- 不保证企业微信传输层的数学意义“恰好一次”。Webhook 没有服务端幂等键；发送成功但响应丢失时仍可能出现不可消除的极少数重复。本文保证应用层相同成交事件不会因周期快照而主动重复发送。
- 不引入事务型通知 outbox。SQLite 提交成功后、调用通知发送前若进程异常退出，该成交在重放时会被账本识别为已存在，因此可能漏掉执行回报；如未来要求“至少一次”交付，需要另立 outbox 设计，不能靠延长冷却或 Markdown 去重补足。
- 不恢复本地模拟盘，也不改变 JoinQuant 模拟盘主数据源职责。

## 4. 方案选择

### 4.1 不采用：延长通知冷却时间

延长冷却只能降低刷屏频率，冷却到期后仍会重复，不解决事件身份错误。

### 4.2 不采用：只按订单摘要生成稳定哈希

`order_id + filled + status` 比完整 Markdown 稳定，但仍把通知事实放在通用 JSON 状态文件中，也难以准确表达一笔订单的多次部分成交。

### 4.3 采用：SQLite 新成交事件驱动

schema version 6 已有不可变 `fills` 表和唯一 `fill_id`。账户快照事务入账时收集本次首次插入的成交，并把这些新增成交作为回调结果返回。通知层只消费本次新增成交；重复成交因唯一约束不再进入通知输入。

## 5. 执行回报数据流

```text
JoinQuant 账户快照
→ 规范化订单和 trades
→ SQLite 事务 upsert 订单
→ INSERT OR VERIFY fills
→ 收集本次 new_fills
→ 完成持仓周期、退出意图和自动对账
→ 事务提交
→ 发布兼容 JSON
→ 仅对 new_fills 发送执行回报
→ 失败时进入现有通知重试队列
```

规则：

- `trades` 存在时，以首次插入的不可变 `fill_id` 为唯一通知事实。
- 同一 `fill_id` 内容一致的重复回调返回“已存在”，不再通知；内容冲突仍按账本完整性错误处理。
- 同一订单产生多个真实成交 ID 时，可以在同一条回报中合并展示，也可以按本次回调分组，但每个成交只属于一个通知批次。
- 50 股部分成交后再成交 50 股，两个不同成交事件均应通知；同一累计 100 股的后续周期快照不得再通知。
- 兼容旧快照且没有 `trades` 时，只在订单累计成交量相对 SQLite 已保存状态真实增加时生成一次 legacy 增量事件；周期快照中未变化的历史订单不得通知。
- 通知发送失败不回滚已提交成交；失败正文和稳定事件键进入现有重试队列，由 `notify_retry.py` 补发。

## 6. 执行回报内容

执行回报展示本次新增成交，而不是快照内全部历史订单：

```text
【JoinQuant 模拟盘执行回报】
服务器时间：YYYY-MM-DD HH:MM:SS Asia/Shanghai
成交时间：YYYY-MM-DD HH:MM:SS
股票、方向、本次成交数量、本次成交价格
订单累计成交/委托数量、订单状态
关联信号或退出原因（存在时）
```

快照总资产和当前现金可以作为辅助信息展示，但不得参与成交事件身份或去重键。

## 7. 全量复盘模型

### 7.1 样本范围

复盘事实源是“企业微信成功发送的买点提醒”。只有 `send_markdown(...)` 返回成功后记录的 `kind=买点` 信号进入买点复盘；风险、卖出和普通扫描摘要不混入买点胜率。

每条记录绑定：

- 成功推送时间和交易日；
- 股票、信号 ID、模式和买点类型；
- 推送价、建议入场、止损、止盈和建议仓位；
- 当时的策略分、市场状态和题材信息。

### 7.2 复盘批次

- D+0：只列出当日成功推送的买点清单，不评价成败。
- D+1：完整复盘上一交易日所有成功推送买点，一个不漏。
- D+3、D+5、D+10：按 A 股交易日计算阶段收益、最大浮盈、最大回撤、是否入场、止盈和止损。
- 股票不再进入候选池也必须复盘。
- 行情缺失时保留该信号并标记“行情缺失”，不得静默删除或跳过。
- 当期信号超过单条微信容量时按固定行数分片，标题显示 `第 N/M 组` 和批次总数；分片不改变完整样本数。

复盘行情必须来自当日完整行情快照中按股票代码取值，不能继续使用仅包含 TopN 候选的结果表。

## 8. 统一服务器时间

所有企业微信消息在统一出口 `WeComNotifier.send_markdown` 增加：

```text
> 服务器时间：YYYY-MM-DD HH:MM:SS Asia/Shanghai
```

时间在实际发送尝试时生成，使用服务器本地时区。调用方提供的去重键在加时间前已经确定，因此时间变化不得破坏现有通知去重。

通知失败队列保存未加动态时间的业务正文；重试时重新增加当前服务器时间，避免出现两个时间行，并让用户看到实际收到消息的发送时间。原事件时间仍由各业务正文独立展示。

## 9. 数据增长与保留

本方案不新增数据库表、JSONL 或逐次报告文件。

- `fills` 已属于长期正式交易账本；新增返回字段只暴露当前事务新插入的少量记录，不复制持久化数据。
- `signal_watchlist.json` 继续原子覆盖。为了覆盖 D+10 交易日，热保留调整为至少 20 个自然日，并设置最多 500 条硬上限。
- 以每日 15 条成功买点估算，20 日约 300 条；单条按 1–2 KB 估算，目标文件低于 1 MB。达到 500 条时按最旧推送时间淘汰，不能优先删除尚未完成 D+10 的较新记录。
- 通知失败队列沿用成功后删除规则，不新增无限追加流。
- `wecom_notify_state.json` 不再为同一成交的每分钟快照产生新键，实际增长将下降。

## 10. 异常与降级

- SQLite 事务失败：保持现有 503、停买/熔断和对账告警语义，不发送无法入账的执行回报。
- 企业微信失败：进入现有重试队列，不改变交易状态。
- 完整行情缺少复盘股票：输出“行情缺失”，保留后续复盘资格。
- 上一交易日没有成功买点：发送简短“D+1 无买点样本”摘要或保持现有盘后摘要，不虚构样本。
- 复盘文件损坏：安全返回空列表并产生可观察警告；不得影响交易、卖出和账户回传。

## 11. 测试要求

### 11.1 执行回报

- 同一成交连续回传 5 次，只发送 1 次执行回报。
- 服务重启后相同成交再次回传，不重复发送。
- 50 股部分成交后新增 50 股，产生两次合法增量回报。
- 同一 `fill_id` 内容冲突继续触发账本错误，不被通知去重掩盖。
- 无 `trades` 的 legacy 快照只在累计成交增加时通知。
- 发送失败进入队列，重试成功后队列移除。

### 11.2 复盘

- 上一交易日所有成功买点均进入 D+1，数量超过 6 时分片但不遗漏。
- 股票掉出当前候选池仍能从完整行情中复盘。
- 当天新信号只进入 D+0，不混入上一交易日 D+1。
- 风险提醒不计入买点复盘。
- 行情缺失明确展示且记录不丢失。
- 周末和配置节假日使用上一 A 股交易日，不按自然日前一天。
- D+3、D+5、D+10 使用交易日序号。

### 11.3 时间与兼容

- 所有消息只有一行服务器时间。
- 时间不参与调用方去重键。
- 失败重试显示重试发送时间并保留原事件时间。
- 现有交易计划、健康告警、控制告警、备份告警和普通扫描通知内容保持兼容。

## 12. 状态提升与验收

- spec、计划、代码和测试完成：`implemented`。
- 服务器代码部署且服务运行：`deployed`。
- 真实交易日出现新增成交只推送一次、D+1 全量复盘和统一时间证据：`observed`。
- 连续 3 个有效交易日无周期快照重复成交回报，D+1 样本与成功买点逐条一致，通知失败重试无丢失：通知幂等可标记 `validated`。
- D+3/D+5/D+10 策略效果仍需对应周期积累，不因通知链路 3 日稳定而提前标记策略有效。

## 13. 权限与部署边界

本设计批准不等于授权推送、部署或重启。代码实现、Git 推送、服务器拉取、服务重启和 JoinQuant 网站模板变更必须分别遵守用户当次授权。Codex 自动审核任务仍只有只读权限，不能补发通知、修改复盘记录或改变交易状态。
