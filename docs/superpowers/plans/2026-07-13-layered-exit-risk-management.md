# 分层退出、买入与动态风险管理实施计划

更新日期：2026-07-14

> 主文档：`docs/project_roadmap.md`。权威业务设计见 `docs/superpowers/specs/2026-07-13-layered-exit-risk-management-design.md`。本文件只保留当前 Batch A-G 唯一实施顺序；早期分批顺序已被本计划替代并移入归档。

## 状态与实施原则

本项目当前使用 JoinQuant 模拟盘。用户已确认在一次开发周期内补齐买入、卖出和执行闭环，达到统一部署门槛后一次同步模拟盘；观察证据仍按模块拆分，以便分别归因执行、回撤和异常。

当前已知状态：

- Batch A-E：已随本地提交 `9f4c12d` 完成 `implemented`，尚未推送、部署、观察或验证。
- Batch F：部署与真实模拟盘观察 `planned`，需要用户当次单独授权。
- Batch G：半自动参数复核 `planned / not implemented / not deployed / not observed / not validated`。
- 本地 SQLite schema version 6、自动备份 CLI/轮转/恢复演练和 systemd 模板不代表服务器已升级、timer 已安装或自动证据已形成。

统一状态模型：本地代码和测试完成是 `implemented`；服务器及 JoinQuant 同步是 `deployed`；有效交易日产生证据是 `observed`；达到门槛并人工验收是 `validated`。

## Batch A：执行安全与成交对账（P0）

**目标：** 任何退出信号都能区分“想卖、已委托、部分成交、无法卖和已完成”，失败后保持退出意图，成交后按实际结果推进状态。

**当前实现范围：**

- JoinQuant 快照回传总数量、可卖数量、冻结和当日数量；服务器不把总持仓等同可卖持仓。
- 下单目标同时受总持仓、可卖数量、冻结数量、100 股单位和未完成委托约束。
- SQLite 保存退出意图、目标数量、委托 ID、状态、成交数量、剩余目标、失败原因和最后对账时间。
- `submitted / partial / filled / cancelled / rejected / T+1 / suspended / limit_down` 使用明确状态并支持重复回调幂等。
- 部分成交或撤单后按剩余目标重算；不可卖、停牌、跌停和临时拒单不消费退出意图。
- 止盈阶段只由确认成交或同步持仓变化推进。
- 当前价异常、陈旧或偏离可信价超过阈值时暂停新动作并报警。

**验证门槛：** 覆盖不可卖、部分可卖、部分成交、撤单重试、跌停连续重试、已有委托和重复快照；连续 3 个有效模拟交易日无重复有效委托、超量卖出、错误完成和状态串仓。

## Batch B：旧持仓迁移与退出统一（P0）

- 迁移报告逐股列出数量、成本、可卖数量、冻结止损、ATR 来源、模式和建仓日期可信度。
- 能关联买入信号的持仓启用完整规则；无法关联的旧持仓只启用已有固定硬止损。
- 人工确认字段只能显式配置，Codex 自动审核员不得填写或修改。
- 清仓再买创建新周期；加仓更新加权成本但不得放宽冻结止损。
- 移动止盈最高价只使用已确认的可信行情，行情缺口期间不更新。

**验证门槛：** 部署前迁移报告持仓集合与 JoinQuant 快照完全一致；任一持仓缺少安全止损即阻止统一部署。

## Batch C：真实组合买入风控（P0）

- 使用真实可用现金、总资产、已有持仓、未完成买单冻结资金、当日新增仓位和真实成交，不使用空组合做最终检查。
- 组合开放风险包含活动持仓与未完成买单；默认 `NORMAL=4%`、`CAUTION=2%`、`RISK_OFF=0%`。
- 候选按最终评分和风险收益质量排序，依次消耗现金与风险容量，缩为 100 股单位后不足一手则拒绝。
- 行业目标市值不超过 25%，同题材不超过 20%，分类缺失时单票不超过 10%。
- 当日新增持仓、订单、换手、连续亏损和账户回撤进入强制层并输出稳定拒绝原因。

**验证门槛：** 多股票同时满足买入时，总额、开放风险、行业/题材集中度和冻结现金均不越界；同一输入重复运行结果一致。

## Batch D：买入可交易性和追价保护（P1）

- 排除停牌、ST/*ST、退市整理、特殊上市阶段和数据不足标的。
- 按交易板块元数据处理交易单位、涨跌幅和价格笼子，不只按代码前缀猜测。
- 同时检查成交额、换手率和最近成交连续性。
- 接近涨停不买；超过计划入场价 `min(2%, 0.5ATR/entry)` 时不追买。
- short/mid 使用不同有效期；过期信号记录明确原因。
- JoinQuant 下单前使用平台当前价格和可用现金再次复核。

## Batch E：市场状态滞后与退出冷却（P1）

- `NORMAL → CAUTION → RISK_OFF` 至少连续 2 轮确认；风险恢复至少连续 3 轮确认。
- `RISK_OFF` 禁止新买，`CAUTION` 只接受高分并将风险预算减半；市场状态不能下移初始止损。
- 硬止损、时间止损、正常止盈和人工退出使用不同冷却原因与期限。
- 状态进入 SQLite 有界表或覆盖式状态，不新增无界 JSONL。

## Batch F：统一部署与分栏观察（P0）

**部署前门槛：**

1. 本地完整测试通过，Linux 服务器补跑 `tests/test_joinquant_linux_script.py`。
2. SQLite 在线备份、恢复副本、`PRAGMA integrity_check` 和 schema migration 通过。
3. 旧持仓迁移报告通过人工审核。
4. 服务器代码、schema version、JoinQuant 网站模板和期望模板版本一致。
5. 各新增能力有独立开关和回滚路径；硬止损不受普通买入健康门关闭。

**观察分栏：**

- 执行正确性：目标、可卖、委托、成交、剩余持仓和重试。
- 卖出质量：硬止损滑点、+2R 后剩余收益、移动止盈回撤和时间止损后表现。
- 买入质量：追价拒绝、现金利用、开放风险、集中度和被拒信号后续表现。
- 市场状态：切换次数、抖动、弱市新增风险和恢复滞后。
- 数据一致性：JSON、SQLite、JoinQuant 快照、委托和健康报告。

首个交易日逐笔人工核对；连续 3 个有效交易日验证执行安全；至少 20 个有效交易日评价策略表现。执行错误立即回滚对应模块，短期亏损本身不触发参数修改。

## Batch G：半自动参数复核（P2）

自动任务只负责统计、生成少量候选和准入报告；用户按候选 ID 与 SHA-256 明确批准后，才允许在另行授权的任务中发布到 JoinQuant 模拟盘。Codex 只读审核员不得批准、改参、部署、重启或回滚。

未满 20 个有效交易日、30 个闭合持仓周期或参数族 30 个相关样本时，只生成“数据不足”报告。20 日只允许开始复核；候选进入可批准列表还需要通过 strict 质量门的完整历史回测、至少 3 个 walk-forward 窗口和 20 个有效模拟盘交易日；缺少可用历史回测证据时替代门槛为至少 60 个有效模拟盘交易日。

每次最多 5 个候选，每个候选只改变一个参数族，并验证扣费净收益、Profit Factor、最大回撤、尾部损失、离群交易依赖和关键分层退化。硬止损存在性、卖出安全、T+1、可卖数量、绝对仓位/开放风险上限、幂等和账本一致性不属于可调范围。

## 唯一实施顺序

```text
Batch A 执行安全与成交对账
→ Batch B 旧持仓迁移与退出统一
→ Batch C 真实组合买入风控
→ Batch D 可交易性与追价保护
→ Batch E 市场状态滞后与冷却
→ Batch F 统一部署和分栏观察
→ Batch G 数据就绪后的半自动参数复核
```

Batch A-E 不能因本地测试通过而标为 `deployed / observed / validated`。Batch F 的提交、推送、服务器操作、网站模板更新和服务重启仍需要用户当次分别授权；Batch G 依赖真实数据，不能在开发阶段伪造完成。
