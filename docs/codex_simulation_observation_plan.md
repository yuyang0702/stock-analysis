## 1. 目标与边界

本方案把 Codex 定位为“只读观察员和阶段评审员”，而不是自动维修或自动部署系统。

Codex 可以：

- 读取服务器健康报告、结构化运行数据和经过授权的日志。
- 对照主文档、从属文档、代码、测试和服务器运行证据判断当前实施阶段。
- 分析系统稳定性、执行质量、数据完整性和策略样本质量。
- 观察主要运行文件、目录、SQLite和WAL的增长速度及预计触达阈值时间。
- 只读检查备份 `status.json`、manifest 元数据、SHA-256、latest/季度报告、目录大小和 systemd timer 状态。
- 检查新增或变更的持久化功能是否符合 `docs/data_storage_policy.md`。
- 在存储优化实施前评估调用方、业务依赖、兼容要求、测试缺口和回滚条件。
- 在存储优化部署后对照优化前基线，判断统计口径、通知语义和交易闭环是否保持不变。
- 输出上午摘要、每日验收、每周阶段报告和优化建议。
- Batch G 实现后，只读解释参数复核报告、候选门槛和活动版本一致性。
- ML-7 实现后，只读解释训练型影子模型的模型版本、时间切分、验证/保留集和模拟盘对照证据。
- 历史回测部署后，只读检查质量报告、运行摘要、数据哈希、walk-forward 和比较契约；不得把 `price_core` 代理结果升级为完整历史回测证据。
- 通过邮件发送重要报告；普通交易通知继续使用企业微信。

Codex 不可以：

- 修改代码、配置、文档或服务器文件。
- 创建 Git 分支、提交、推送、合并或改写历史。
- 部署服务器、重启服务、修改 systemd 或运行升级命令。
- 修改策略、评分、仓位、止盈止损或风控阈值。
- 创建、撤销、补发或修改任何订单和持仓。
- 因日志异常自行改变交易状态。
- 移动、压缩、归档、清理或删除任何服务器运行数据。
- 运行 `backup`、`backup-drill`、恢复、轮转、清理或主库替换命令。
- 导入历史数据、启动历史回测/比较、固定或清理历史运行、批准参数，或用自动审核任务生成新的回测证据。
- 创建参数候选、写入批准/拒绝决定、激活或回滚参数版本。
- 批准、发布、替换或回滚机器学习模型。

任何优化建议只有在用户另行明确授权后，才进入独立的设计或实施任务。

## 2. 复用现有能力

服务器实时异常发现已经实现，本方案不重复建设第二套监控系统。

现有能力包括：

- `stock-joinquant-health.timer` 每 5 分钟运行健康检查。
- `joinquant_health.py` 检查信号、账户快照、API 访问、订单失败、持仓一致性、模板版本、SQLite 和 JSON/SQLite 一致性。
- `output/joinquant_health_YYYYMMDD.md` 保存当日健康报告。
- `cache/joinquant/health_history.jsonl` 保存连续健康检查记录。
- `cache/joinquant/api_events.jsonl` 保存信号拉取、latest、快照回传和 API 异常事件。
- `cache/joinquant/signals.json` 保存当前 JoinQuant 信号计划。
- JoinQuant 账户快照和持仓同步数据保存实际账户、持仓和订单回报。
- `cache/trading/trading.db` 保存 Batch 1 策略运行、信号和观察型风控账本。
- 企业微信负责交易通知、成交回报和严重健康异常报警。
- 通知失败队列和 timer 负责企业微信失败重试。

`origin/main` 中的 `trading_backup.py`、7/4/12 轮转、隔离恢复演练和 systemd 模板已 `implemented（已推送）`，服务器当前代码 `52b3653` 已包含这些能力；timer 是否安装及其连续运行证据仍待只读核验。Codex 只能读取备份状态、manifest、报告和 timer 证据；不得因具备服务器只读 SSH 权限而执行备份或恢复命令。

Codex 的新增价值不是“发现有没有 stale”，而是把已有证据转化为：

1. 当前功能处于 planned、implemented、deployed、observed 还是 validated。
2. 当前交易日是否构成有效观察样本。
3. 阶段 1 是否达标，是否具备进入下一阶段的条件。
4. 下一步最值得优化的内容是什么，是否阻塞阶段推进。

## 3. 阶段判断证据链

每次阶段判断必须同时核对四类证据，不能只依据单段日志或某个代码文件。

### 3.1 文档证据

- `docs/project_roadmap.md`：唯一主文档，定义业务方向、阶段、状态和优先级。
- 主文档“当前有效从文档索引”：定义哪些从文档仍是当前依据及其读取条件。
- 只读取索引中与本次审核范围相关的执行方案、专项设计和实施计划，不默认遍历全部 `specs/` 或 `plans/`。
- `docs/archive/`：仅在追溯历史决策时读取，不参与当前状态、阶段、优先级或验收结论。

文档中“计划、待实现、部分实现、已废弃、暂不启用”不能被误判为线上已完成能力。

### 3.2 代码证据

- 功能入口是否存在并接入主流程。
- 配置默认值和环境变量是否生效。
- 数据结构、异常处理和兼容路径是否完整。
- 自动化测试是否覆盖关键正向、失败和边界路径。
- systemd/service/timer 生成脚本是否包含对应任务。
- Git 提交历史是否能说明功能何时进入主分支。

代码存在只表示 implemented，不表示 deployed 或 validated。

### 3.3 部署证据

- 服务器当前 Git SHA 是否包含目标功能。
- 服务和 timer 是否启用并处于正确状态。
- 私有配置是否加载，但报告不得泄露 token、webhook、SMTP 授权码等秘密值。
- SQLite schema、文件路径和报告输出是否符合当前版本。

### 3.4 运行证据

- 功能是否经历真实 A 股交易日和完整交易时段。
- 信号、拉取、委托、成交、回传、同步和通知是否形成闭环。
- 是否积累足够天数和样本量。
- 异常是否可解释、可重复定位，并且没有破坏数据一致性。

## 4. 统一状态模型

每项能力使用以下状态：

| 状态 | 含义 |
| --- | --- |
| `planned` | 主从文档已规划，代码尚未完成。 |
| `implemented` | 代码和测试已完成，尚未确认服务器部署。 |
| `deployed` | 服务器已运行包含该功能的版本。 |
| `observed` | 已在真实模拟盘交易日产生运行证据。 |
| `validated` | 达到主文档规定的连续天数、成功率、一致性和样本标准。 |

## 5. 定时分析任务

第一版仅建立两个日常任务；每周任务在日常报告连续运行 3 个交易日且质量可接受后再启用。

### 5.1 上午摘要：交易日 11:35

检查范围限定为当日 00:00 至 11:35，优先读取结构化文件，仅在发现异常时读取相关日志片段。

检查项：

- 核心服务和 timer 状态。
- 上午策略扫描轮次、完成时间和异常。
- 最新健康状态、稳定性评分和 issue codes。
- SQLite schema、健康、策略运行数、信号数、风控记录数。
- 当前 JSON 信号数与 SQLite 当前信号内容一致性。
- JoinQuant signals/latest 拉取次数和 API 错误。
- 账户快照回传次数、时效和模板版本。
- 委托、成交、失败、跳过及原因。
- 服务器与 JoinQuant 持仓一致性。
- 企业微信失败队列是否增长。
- 未捕获异常、数据库锁、序列化错误和重复订单迹象。

输出模板：

```text
上午运行：正常 / 需观察 / 异常
证据时间范围：YYYY-MM-DD 00:00-11:35
系统稳定性：……
策略扫描：……
SQLite 与数据一致性：……
JoinQuant 通信与执行：……
账户和持仓：……
发现的问题：……
下午重点关注：……
是否需要人工处理：是 / 否
```

### 5.2 全天验收：交易日 15:40

检查完整链路：

```text
策略扫描 -> 候选/评分 -> 信号 -> SQLite -> JSON -> JoinQuant 拉取
-> 委托/成交 -> 账户快照 -> 持仓同步 -> 微信通知 -> 复盘数据
```

至少统计：

- 扫描轮次、候选数、信号数和过滤原因。
- 买入、卖出、委托、部分成交、全部成交、失败和跳过数量。
- 信号拉取、latest 请求、快照回传和 API 异常次数。
- 信号到拉取、拉取到委托、成交到快照的延迟。
- 失败订单原因是否符合市场规则或系统预期。
- 重复信号、疑似重复委托和不一致事件。
- JSON/SQLite 一致性和 SQLite 健康。
- 账户总资产、现金、持仓数量和持仓一致性。
- 企业微信实际成交通知是否与 SQLite 当日新增 fill/legacy 累计成交增量一致，周期快照是否产生重复回报。
- 所有企业微信消息是否恰有一行服务器发送时间；失败重试是否保留业务事件时间并显示新的发送时间。
- D+1 样本数是否与上一交易日成功推送的买点逐条一致，D+0/D+3/D+5/D+10 是否按交易日分批，行情缺失和掉出候选池是否仍保留。
- 当日策略样本、D+N 标签更新和可用复盘数据。

输出模板：

```text
今日结论：通过 / 有条件通过 / 不通过
有效观察日：是 / 否，原因……
连续稳定日：N/10
当前阶段矩阵：……
系统稳定性：……
执行质量：……
数据完整性：……
策略样本质量：……
P0/P1 问题：……
优化建议：……
下一阶段建议：保持当前阶段 / 有条件准备 / 可以推进
```

### 5.3 每周阶段报告：周五 16:00（第二批启用）

每周报告基于每日结构化结论，避免重新读取整周原始日志。

内容包括：

- 有效观察日和连续稳定日。
- 信号、委托、成交、失败和成交率趋势。
- D+1/D+3/D+5、最大浮盈、最大回撤、胜率和盈亏比。
- 原策略与影子评分对照。
- 不同市场状态、行业和题材分组表现。
- API、快照、持仓、SQLite 和通知异常趋势。
- 本周新增、持续和已关闭问题。
- 阶段完成度与下一周唯一优先验证假设。

## 6. 有效观察日判定

以下条件同时满足时，当日才计入阶段 1 连续观察：

- 当日为 A 股交易日。
- 核心策略和 API 服务覆盖主要交易时段。
- 至少有预期的扫描和健康检查记录。
- JoinQuant 发生预期的信号拉取和账户快照回传。
- SQLite 可用，JSON/SQLite 一致性可计算。
- 当日证据没有因日志缺失、时间错误或版本不一致而失真。

没有买入或成交并不自动使当日无效。只要链路正常、没有符合条件的信号或委托，可以作为“无交易但系统正常”的有效观察日。

## 7. 阶段 1 验收规则

主文档当前要求至少连续 10 个有效交易日观察。

### 7.1 2026-07-13 新退出策略观察项

Codex 自动审查仍为只读审核员，不得运行完整对账、执行 `trading_control.py`、进入解锁向导、修改控制状态，也不得修改代码、配置、数据库、JoinQuant 策略、委托或服务状态。新版本部署后可以只读查询 schema version 7、最新对账摘要、差异原因码、控制状态和恢复资格，并继续核对同股唯一动作、稳定退出ID、`target_qty`、可卖数量、部分成交、止盈阶段、硬止损不下移、组合开放风险、集中度、追价过滤和市场状态滞后。报告必须分别展示 `implemented / deployed / observed / validated`；`e2ce5b5` 的服务器部署证据和用户确认的网站模板更新可作为 `deployed` 依据，真实交易日行为及任何未被新证据覆盖的外部状态仍只能报告“待观察/待验证”。
本门槛用于阶段 1 基础系统稳定性验收。`docs/superpowers/specs/2026-07-11-simulation-stability-ledger-design.md` 中的 20 个有效交易日用于完整账本加固与策略验证；10 日通过不等于 20 日专项完成。非交易日 readiness 结论仅表示静态结构检查，不构成有效观察日、阶段放行或实盘准入证据。

### 7.2 必须满足

- 连续 10 个有效交易日无不可解释的核心服务中断。
- 信号拉取成功率达到主从文档要求，当前目标为 `>= 99%`。
- 账户快照每日稳定回传。
- 重复委托为 0。
- JSON/SQLite 不一致为 0。
- 持仓不一致为 0，或每次差异均有明确且可接受的时序解释。
- 失败订单原因可分类、可追踪、可解释。
- 企业微信异常报警可收到或由重试队列补发。
- JoinQuant 网站模板版本与服务器期望一致。

### 7.3 不应混为同一结论

- 系统稳定，不等于策略有效。
- 单日盈利，不等于可以进入实盘。
- 没有成交，不等于链路失败。
- 代码已实现，不等于交易日验证完成。
- 观察型风控有记录，不等于实盘级强制风控完成。

## 8. 策略验证指标

策略评价至少分为执行质量和策略效果两层。

执行质量：

- 信号发布率、拉取率、委托率和成交率。
- 平均和分位数延迟。
- 部分成交比例、失败原因和滑点。
- 卖出链路可靠性。

策略效果：

- 有效买入样本数。
- D+1、D+3、D+5 收益。
- 最大浮盈、最大回撤、胜率和盈亏比。
- 止盈、止损和未入场比例。
- 不同市场状态、行业和题材表现。
- 原策略 TopN 与影子评分 TopN 对照。

样本不足时必须明确写“证据不足”，不得用少量交易推导稳定收益结论。

## 9. 问题与优化建议分级

| 级别 | 定义 | 是否阻塞阶段推进 |
| --- | --- | --- |
| P0 | 交易正确性、重复订单、账本损坏、严重持仓错误。 | 是 |
| P1 | 数据完整性、关键链路稳定性、长期验证可信度问题。 | 通常是 |
| P2 | 策略评价、统计质量、性能或功能增强。 | 通常否 |
| P3 | 展示、体验、文档或低风险运维优化。 | 否 |

每条建议必须包含：

```text
编号
级别
问题
证据
影响
建议
预期收益
修改风险
是否阻塞下一阶段
```

报告不得直接实施建议。用户需要另行明确指定要实施的编号。

### 9.1 半自动参数复核审核（planned）

此职责只有在 `docs/superpowers/specs/2026-07-14-semi-automatic-parameter-review-design.md` 对应代码部署后才生效；当前没有可供 Codex 审核的候选登记和批准系统。

届时 Codex 可以只读核对：

- 是否达到 20 个有效交易日、30 个闭合持仓周期和参数族样本门槛。
- 是否按时间切分并隔离保留集，是否存在未来数据或参数版本混用。
- 候选相对基线的净收益、Profit Factor、最大回撤、尾部损失、成本、离群交易依赖和分层退化。
- 候选 ID、内容哈希、父版本、人工决定、活动版本、服务器 Git SHA 和 JoinQuant 模板版本是否一致。
- 当前能力究竟处于 `planned / implemented / deployed / observed / validated` 哪一状态。

Codex 输出只能是“数据不足、拒绝建议、可供人工评审、保持观察或建议回滚评审”等证据化结论。定时任务不得调用管理命令，不得把候选推进到 `approved`，不得因报告显示通过而自动改参、提交、推送、部署、重启或操作交易。用户的历史授权也不能作为下一次发布授权。

### 9.2 数据存储治理审核

Codex审核员同时承担只读的数据存储治理职责，但不执行归档或清理。

日常报告至少观察：

- 磁盘使用率和可用空间。
- `health_history`、`api_events`、账户快照历史和持仓事件文件大小。
- `output/`文件数量、总大小和每日新增量。
- `scan_cache.json`大小和增长趋势。
- SQLite主库、WAL、SHM大小和健康状态。
- 策略样本文件大小和标签更新耗时。
- 最后成功备份、SHA-256/完整性检查、轮转异常、季度恢复演练和归档时间；备份能力部署前必须明确标为“无服务器证据”。

存储章节输出格式：

```text
存储健康：正常 / 需观察 / 异常
主要增长对象：……
当日增长量：……
近N日平均增长：……
预计触达告警阈值：……
规范符合性：通过 / 部分通过 / 不通过
是否影响当前阶段：是 / 否
建议：……
```

当用户准备实施存储优化时，Codex先生成变更前影响评估：

- 涉及文件、写入方和读取方。
- 是否被健康检查、ML标签、持仓同步、通知或阶段报告依赖。
- 是否改变文件路径、格式、字段、计数或时间范围。
- 旧数据兼容和迁移要求。
- 优化前基线指标。
- 自动化测试和真实交易日观察要求。
- 明确回滚条件。

优化部署后，Codex至少对照一个完整交易日：

| 指标 | 优化前 | 优化后 | 结论 |
| --- | ---: | ---: | --- |
| 健康检查次数与耗时 |  |  |  |
| 信号拉取次数 |  |  |  |
| 快照回传次数 |  |  |  |
| API异常和失败订单 |  |  |  |
| JSON/SQLite一致性 |  |  |  |
| 持仓一致性 |  |  |  |
| 微信报警和静默语义 |  |  |  |
| 文件增长量 |  |  |  |

只有业务计数、健康语义、非交易时段静默规则和交易闭环保持一致，且存储性能达到预期时，Codex才建议继续下一批优化。Codex无权自行回滚、继续部署或扩大改动范围。

### 9.3 训练型影子模型审核（planned）

此职责只有在 `docs/superpowers/specs/2026-07-15-trained-shadow-model-design.md` 对应代码部署后才生效；当前只有规则型 `shadow_score.py`，没有可供审核的训练模型或模型权限状态。

届时 Codex 可以只读核对：

- 每5分钟候选批次、标签成熟度、数据缺口、精确 `available_at <= decision_at` 和版本一致性；

## 10. 下一阶段结论

### 保持当前阶段

出现任一情况：

- 存在未关闭 P0/P1。
- 连续有效交易日不足。
- 重复订单、账本不一致或持仓不一致不可解释。
- 关键运行证据缺失。
- 策略样本不足以支持下一阶段目标。

### 有条件准备

- 阶段 1 系统链路基本稳定。
- 可以开始准备下一阶段的设计、数据或离线工作。
- 仍保持 JoinQuant 模拟盘主流程，不宣告阶段验收完成。

### 可以推进

- 主文档阶段 1 所有硬性标准满足。
- 没有未关闭 P0/P1。
- 运行证据、样本和报告足以支撑下一阶段目标。
- 报告列出仍需保留的模拟盘并行观察项。

Codex提供证据化建议，最终阶段决策仍由用户确认。

## 11. 访问与安全

本地计划任务需要 Windows 开机、联网、Codex 桌面应用运行且项目目录可用。

服务器访问建议创建只读用户 `stockmonitor`，只允许读取：

- 健康报告和健康历史。
- API 事件、当前信号、账户快照和必要日志摘要。
- SQLite 只读查询结果。
- 当前服务器 Git SHA 和服务状态。

禁止该用户：

- sudo、systemctl 写操作和服务重启。
- 修改项目文件或运行数据。
- Git pull、commit、push、merge。
- 读取 `stock-analysis.env` 的秘密值。

SSH 密钥应为专用只读用户密钥，不使用 root 免密密钥。

## 12. 邮件报告

邮件用于：

- 11:35 上午摘要。
- 15:40 全天验收。
- 周五阶段报告。
- P0 严重问题摘要。

企业微信继续用于交易计划、成交回报和现有健康异常报警。

SMTP 授权码只保存在服务器私有配置中，不进入 Git、报告或日志。第一版可以先在 Codex 任务中生成报告，确认内容稳定后再接入邮件，避免同时引入过多变量。

## 13. Token 控制

- 优先读取当日结构化报告和计数，不读取全部历史日志。
- 只有出现异常代码时才读取相关时间窗口日志。
- 上午任务只读取上午增量；全天任务读取当日摘要。
- 每周任务复用每日结论，不重新分析整周原始日志。
- 相同 issue code 和堆栈摘要去重。

预计正常交易日：

- 上午摘要：5,000-15,000 tokens。
- 全天验收：8,000-25,000 tokens。
- 合计：13,000-40,000 tokens。

实际消耗取决于日志量、异常数量、模型和推理强度，应在前三个交易日记录后再调整。

## 14. 实施步骤

### 第一批：只读分析验证

1. 创建服务器只读用户和专用 SSH 密钥。
2. 手工执行一次只读数据采集，确认不泄露秘密值。
3. 手工生成一份上午报告和一份全天报告。
4. 校准有效观察日、成功率和异常分类口径。
5. 创建交易日 11:35 和 15:40 两个本地计划任务。
6. 连续运行 3 个交易日，检查报告质量、误报和 Token 消耗。

### 第二批：稳定化

7. 固化每日结构化结论和连续稳定日计数。
8. 增加周五 16:00 周报。
9. 接入 SMTP 邮件报告。
10. 连续观察到主文档要求的 10 个有效交易日。

### 暂不实施

- Codex 自动修改代码。
- 自动 Git 提交、推送、合并。
- 自动服务器部署或重启。
- 审批网页和审批链接。
- 任何自动交易状态变更。
- 参数候选的自动批准、自动激活或自动回滚。

## 15. 自动化任务与结果窗口

建议任务名称：

- `量化系统上午运行分析`
- `量化系统每日阶段验收`
- 第二批再增加 `量化系统每周阶段报告`

定时任务作为独立计划任务运行，每次执行产生单独运行结果，不持续污染当前开发对话。任务可以暂停、恢复、修改或删除；删除任务后不再运行。历史运行结果的具体清理能力以 Codex 客户端当时提供的界面为准。
