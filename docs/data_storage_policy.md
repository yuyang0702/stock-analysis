# 数据存储、文件增长与保留规范

> 主文档：`docs/project_roadmap.md`。本文是项目所有运行数据、日志、缓存、快照、报告和数据库的长期开发约束；如业务状态与主文档冲突，以主文档为准。

> 2026-07-15 本地 schema 7 增量：`signals` 与 `exit_intents` 增加生命周期时间列，`execution_issue_state` 按问题对象保存一行最高严重度的当前/恢复状态及最近成功通知时间，`system_state` 保存一个有界的 ERROR 对账自动恢复所有权标记。它不逐分钟追加重复问题事件；CRITICAL 不持有自动恢复所有权，粘性 CRITICAL 只能由有审计原因的人工恢复确认处理。订单、成交、对账差异和控制事件继续作为长期证据，高频账户/持仓仍执行366天热保留。schema 6 备份可恢复后幂等迁移到7。当前为 `implemented（本地工作树） / not deployed / not observed / not validated`。

## 1. 目标

本规范用于防止项目随着长期运行出现以下问题：

- JSON、JSONL、日志、快照和报告无限增长。
- 每次运行永久生成新文件，导致目录中出现数万小文件。
- 定时任务反复读取全部历史，耗时和内存随运行年限增长。
- 为更新一条记录而整体读取并重写大型文件。
- 重复保存没有变化的完整账户、持仓或订单快照。
- 运行数据散落在 Git 仓库根目录并污染工作区。
- 清理文件时误删交易账本、策略样本或审计证据。
- 换服务器时不知道哪些文件需要迁移和恢复。

本规范只约束存储生命周期，不改变策略、交易、评分、仓位、止盈止损、风控和 JoinQuant 执行语义。

## 2. 完成定义

任何新增或修改的持久化功能，只有同时完成以下内容才算完成：

```text
业务功能
+ 数据分类
+ 增长估算
+ 保留期限
+ 轮转/压缩/清理规则
+ 高效查询边界
+ 备份与恢复策略
+ 敏感信息处理
+ 监控指标
+ 自动化测试
```

评审时必须回答：

1. 为什么需要保存这份数据？
2. 它属于当前状态、历史事件、交易账本、缓存还是人工报告？
3. 是覆盖、追加还是增量更新？
4. 每日、每月和每年预计产生多少条、多少文件、多少字节？
5. 在线查询需要多长时间范围？
6. 是否需要长期保留，谁会使用？
7. 何时轮转、压缩、归档或删除？
8. 清理是否支持 dry-run、可重复执行和失败恢复？
9. 是否更适合 SQLite，而不是 JSON/JSONL？
10. 是否包含 token、webhook、授权码、账户信息或其他敏感数据？
11. 换服务器时是否需要迁移？
12. 如何证明跨月、文件损坏、重复运行和并发写入不会丢数据？

## 3. 数据分类

### 3.1 当前状态

示例：

- `cache/joinquant/signals.json`
- 最新账户快照
- 最新持仓文件
- 当前健康报告

规则：

- 使用原子覆盖写，不无限追加历史。
- 文件始终表示最新可信状态。
- 如需历史，写入单独的事件流或数据库。
- 写入失败不得留下半截 JSON。

### 3.2 交易与审计账本

示例：

- `cache/trading/trading.db`
- 后续订单、成交、持仓、权益和对账表

规则：

- 优先使用 SQLite 事务、唯一约束、外键和索引。
- 长期保留，不按普通缓存清理。
- 必须有在线备份、完整性检查和恢复演练。
- 任何删除或压缩必须基于明确的审计和策略验证要求。

Batch G 半自动参数复核计划使用同一 SQLite 保存不可变参数版本、聚合评价、人工决定和激活/回滚事件。该范围当前仅为 `planned`，不属于服务器 schema version 1 或本地 schema version 6。报告不是审批事实源，`output/parameter_review_latest.md` 只允许原子覆盖；不得为每轮扫描或每个参数组合新增 JSONL/独立文件。

### 3.3 事件历史

示例：

- 健康检查历史
- API访问事件
- 账户快照历史
- 持仓网页事件

规则：

- JSONL必须按月或按大小轮转。
- 高频任务只能读取当天、当前月或文件尾部，不得每次读取全部历史。
- 相同高频状态应摘要或去重，完整证据只在变化、成交或异常时保存。
- 旧月份压缩归档，并设置明确保留期限。

### 3.4 临时缓存

示例：

- 行情历史缓存
- 新闻缓存
- 板块与海外市场上下文
- 行业映射缓存
- `scan_cache.json`

规则：

- 必须设置 TTL、最大条目数或最大字节数，至少具备其中两项。
- 不同生命周期的数据不得长期混在同一个大型 JSON 中。
- 缓存损坏或被删除后，主流程应能重建或安全降级。
- 缓存不能作为唯一交易审计证据。

### 3.5 人工报告和导出文件

示例：

- 扫描CSV和Markdown
- 健康日报
- readiness、回测和策略对照报告

规则：

- 盘中高频报告短期保留。
- 每日最终报告和阶段报告长期保留。
- 同一轮结果避免生成多个内容重复的永久文件。
- 大量小文件应按日期分目录或按月压缩。

## 4. 默认目录规范

```text
cache/
├── joinquant/       JoinQuant当前状态和短期事件
├── trading/         SQLite交易与审计账本
├── backtest/        独立历史数据、质量门和回测运行库
├── ml/              策略样本和标签
├── market/          市场上下文缓存
├── industry/        行业映射运行数据
├── portfolio_web/   持仓网页状态和事件
└── archive/         可选的本地压缩归档

output/
├── daily/           每日最终报告
├── scans/           短期盘中扫描结果
└── archive/         历史报告归档
```

运行数据不得写入项目根目录。秘密值只允许存在于 `stock-analysis.env` 或受控秘密管理系统中，不得写入 Git、报告或日志。

## 5. 禁止模式

以下设计默认不允许合入：

- 无轮转和保留策略的无限 `open(..., "a")`。
- 每分钟或每5分钟读取整个多年JSONL历史。
- 每轮扫描永久创建新文件但没有清理规则。
- 每分钟重复追加完全相同的完整持仓和订单快照。
- 使用单一大型JSON保存不同TTL的所有缓存。
- 为修改一条标签而读取并重写整个长期历史文件。
- 在循环中反复解析同一个大型历史文件。
- 将异常响应正文、token、webhook或授权码写入日志。
- 将运行数据放在Git仓库根目录或提交到GitHub。
- 清理脚本直接删除文件，且不支持 dry-run、范围验证和审计报告。

如果确有必要采用例外，专项设计必须解释原因、容量上限、替代方案和回滚方式。

## 6. 默认保留策略

| 数据 | 热数据期限 | 归档期限 | 长期策略 |
| --- | ---: | ---: | --- |
| 健康历史 | 180天 | 1-2年压缩 | 月度归档 |
| API事件 | 90-180天 | 1年压缩 | 异常摘要可长期 |
| 完整账户快照 | 30-90天 | 1年压缩 | 成交和变化节点长期 |
| 账户权益摘要 | 长期 | 可压缩 | 长期保留 |
| 盘中扫描文件 | 30天 | 180天压缩 | 每日最终版长期 |
| 策略信号样本 | 长期 | 定期备份 | 逐步迁入SQLite |
| SQLite交易账本 | 长期 | 多级备份 | 不按日志清理 |
| 历史回测库 | 长期、可重建 | 独立备份或从有版本的数据源重建 | 年度目标低于2GB；达到3GB导入上限即拒绝，不自动删除；不得写入正式交易账本 |
| 参数版本、评价和人工决定 | 长期 | 随SQLite多级备份 | 低频审计数据；每次最多5个候选，不复制全量行情/订单 |
| 参数复核latest报告 | 当前版本 | 无需单独归档 | 原子覆盖；状态变化和阶段验收才生成低频归档 |
| 通知失败队列 | 成功后移除 | 保存失败摘要 | 不无限保留正文 |
| 信号复盘观察池 | 20个自然日、最多500条 | 无 | `signal_watchlist.json` 原子覆盖，目标低于1 MB；行情缺失不提前删除样本 |
| 行业和市场缓存 | 按TTL | 少量备份 | 可重建 |

具体模块可以采用更严格规则；采用更宽松规则必须在专项设计中说明理由。

## 7. JSONL轮转规范

推荐命名：

```text
health_YYYYMM.jsonl
api_events_YYYYMM.jsonl
account_snapshots_YYYYMM.jsonl
```

要求：

- 写入时自动选择当前月份文件。
- 跨月自动创建新文件。
- 当前任务只读取满足时间范围的文件。
- 历史查询最多打开涉及月份，不扫描无关历史。
- 压缩只处理已经关闭的月份，不处理当前写入文件。
- 压缩文件使用 `.jsonl.gz`，保留原始UTF-8 JSONL语义。
- 迁移期间兼容旧单文件，但新记录不得继续无限写入旧文件。

如单月文件仍可能过大，应增加按大小轮转，例如单文件上限 `256MB` 或 `500MB`。

## 8. 账户快照规范

账户数据分为：

1. 最新完整快照：原子覆盖，供同步和展示使用。
2. 高频权益摘要：时间、总资产、现金、持仓数、订单数和关键哈希。
3. 状态变化快照：持仓、订单、成交、模板版本或健康状态变化时保存完整证据。
4. 固定检查点：即使无变化，也可每小时保留一次完整快照。

禁止每分钟长期重复保存完全相同的订单历史。订单事件应使用稳定ID并增量保存，避免在每个账户快照中重复复制全部订单。

## 9. SQLite适用规则

以下情况优先使用SQLite：

- 需要按ID更新单条记录。
- 需要唯一性、幂等、事务或关联查询。
- 需要按日期、股票、信号、订单或状态筛选。
- 数据需要长期保留并持续增长。
- 定时任务需要高效统计而不是全文件扫描。

SQLite要求：

- schema变更通过显式migration。
- 常用查询字段建立索引。
- 写操作使用事务和合理busy timeout。
- WAL大小纳入监控并定期checkpoint。
- 每日使用SQLite在线备份API生成一致性备份。
- 定期运行 `PRAGMA integrity_check` 和 `PRAGMA optimize`。
- 大型 `VACUUM` 不在交易时段运行。

## 10. 备份与恢复

建议备份目录位于项目目录之外：

```text
/opt/stock-analysis-backups/
```

SQLite默认备份保留：

- 最近7份每日备份。
- 最近4份每周备份。
- 最近12份月度备份。

重要缓存、策略样本和私有配置应有迁移清单。备份成功不代表可恢复，至少每季度进行一次恢复演练，并记录：

- 备份SHA256。
- SQLite完整性检查。
- 恢复目标路径。
- 恢复后schema和核心计数。
- 演练时间和结果。

## 11. 统一归档与清理

后续建议实现统一入口：

```bash
python data_retention.py --status
python data_retention.py --dry-run
python data_retention.py --apply
```

要求：

- `--dry-run`显示将压缩、移动或删除的精确文件和预计释放空间。
- `--apply`只能处理白名单目录，必须验证解析后的绝对路径位于项目数据目录内。
- 不处理当前正在写入的文件。
- 不删除SQLite主库、私有配置和未确认备份。
- 重复执行不会重复压缩、破坏或报错。
- 每次执行生成审计报告，例如 `output/data_retention_YYYYMMDD.md`。
- 失败时保留原文件，不留下半截归档。

## 12. 健康监控指标

健康报告后续应逐步加入：

- 磁盘使用率。
- `health_history`、`api_events`和快照历史大小。
- `output/`文件数量和总大小。
- `scan_cache.json`大小和条目数。
- SQLite主库、WAL和SHM大小。
- 最后成功归档时间。
- 最后成功备份时间。
- 归档和备份连续失败次数。

建议阈值：

```text
磁盘使用率 >= 70%：warning
磁盘使用率 >= 85%：critical
单个JSONL >= 500MB：warning
scan_cache >= 100MB：warning
SQLite WAL持续异常增长：warning
连续7天没有成功归档：warning
```

存储告警只用于运维，不直接改变买卖或风控状态。

## 13. 测试要求

新增或修改持久化逻辑至少测试：

- 跨日和跨月轮转。
- 旧路径或旧单文件兼容读取。
- 重复执行的幂等性。
- 空文件、截断文件和损坏JSON处理。
- 并发写入或SQLite锁定。
- 原子替换失败。
- 保留边界日期。
- dry-run不修改任何文件。
- 清理目标无法逃逸白名单目录。
- 压缩失败时原文件仍完整。
- 敏感字段不会进入日志和报告。
- 数据计数、哈希或抽样对账保持一致。

## 14. 当前文件治理优先级

### 14.1 2026-07-13 持仓周期账本增量

当前 `origin/main` 的 SQLite schema version 6代码新增 `orders`、`fills`、`account_snapshots`、`position_snapshots`、`daily_equity`、`reconciliation_runs`、`reconciliation_items`和`control_events`，并保留 schema 5 的持仓周期、委托事件、退出意图和冷却。账户摘要约6万行/年；持仓明细只在状态变化、整点和每日首次收盘检查点保存；年度新增连索引目标低于200MB，超过300MB/年或2MB/日必须告警并暂停新增高频明细。

高频账户摘要保留366天热数据；订单、逐笔成交、日权益、控制事件、异常对账及其引用快照长期保留。清理每日最多运行一次，只删除超过热保留期且没有异常对账引用的账户快照与无差异摘要；未来压缩或扩大清理范围必须另行设计、dry-run并人工授权。

### Batch A：低风险性能优化

1. 健康历史按月轮转。
2. API事件按月轮转。
3. 健康检查只读取当天或当前月份数据。
4. 兼容旧文件并增加测试。

### Batch B：账户快照优化

1. 保留最新完整快照。
2. 增加高频轻量摘要。
3. 持仓、订单、成交或健康状态变化时才追加完整快照。
4. 确认订单标签、健康统计和持仓同步语义不变。

### Batch C：扫描输出和缓存治理

1. 扫描文件按日期组织。
2. 盘中结果短期保留，每日最终结果长期保留。
3. 增加统一归档dry-run。
4. `scan_cache.json`增加TTL、条目数和容量上限。

### Batch D：策略样本SQLite化

1. JSONL与SQLite双写。
2. 唯一性和内容一致性检查。
3. 报告优先读取SQLite。
4. 稳定观察后再决定是否停止JSONL主写。

## 15. 评审清单

每个PR或实施任务涉及持久化时必须确认：

- [ ] 数据分类正确。
- [ ] 路径位于 `cache/`、`output/`或批准的项目外备份目录。
- [ ] 没有秘密值或不必要的个人/账户信息。
- [ ] 有每日和年度增长估算。
- [ ] 有文件数和单文件大小上限。
- [ ] 有保留、轮转、压缩或清理规则。
- [ ] 高频读取不会扫描全部历史。
- [ ] 需要增量更新的数据使用SQLite或等价索引存储。
- [ ] 有备份和恢复说明。
- [ ] 有监控指标和告警阈值。
- [ ] 有跨月、损坏、并发、幂等和清理安全测试。
- [ ] 不改变策略、交易或风控语义。

未满足清单时，功能状态最多标记为“部分实现”，不得标记为“已完成”。
